# Plan: Delta-Snapshot Serialization with Store Reconciliation

## Motivation
Cut the worker→main `postMessage` structured-clone cost by shipping sparse `updated[]`/`removed[]` entity deltas instead of full arrays every frame, while keeping `SnapshotStore` exposing full `snapshot.enemies`/`snapshot.towers` so no render manager changes. Worker allocation is minimized via allocation-free comparison. Secondary motivation: **prepare the snapshot stream for multiple consumers** — the reconciled store becomes the single place that materializes full state from deltas, so future consumers (e.g. a replay recorder, a network relay, analytics) can subscribe to either the raw delta wire stream or the reconciled full snapshot.

## Wire / type changes — `src/sim/SimulationSnapshot.ts`
- Add a `WireSnapshot` discriminated union (used **only** on the worker→main boundary):
  - `entityMode: "full" | "delta"`.
  - **Shared base (`WireCommon`)** — both union members carry every non-entity field that ships every frame in BOTH modes, defined once to avoid drift:
    - `schemaVersion: number`, `frameId: number`, `lastAppliedCommandId: number`, `meta: SnapshotMeta`,
    - `projectiles: ProjectileSnapshot[]`, `particleSpawns: ParticleSpawnRequest[] | undefined`, `spawnStates: SpawnStateSnapshot[]`,
    - `paths`, `pathsVersion`, `waveGraphDots`, `waveGraphDotsGeneration`, `gridLayout`,
    - `lightningEffects`, `stunEffects`.
    These mirror the existing `SimulationSnapshot` non-entity fields exactly (the worker already computes them in `buildSnapshot` — `SnapshotSerializer.ts:24,31-70,79-94`); they are simply threaded through unchanged in both modes. `SvgGameRoot.vue:311` already reads `msg.snapshot.particleSpawns` off the wire, so it must keep compiling — hence the shared base.
  - **Full** (`extends WireCommon`): `enemies: EnemySnapshot[]`, `towers: TowerSnapshot[]` (as today); delta fields absent.
  - **Delta** (`extends WireCommon`): `enemiesUpdated: EnemySnapshot[]`, `enemiesRemoved: number[]`, `towersUpdated: TowerSnapshot[]`, `towersRemoved: string[]`; `enemies`/`towers` absent. Also ship compact **authoritative id-order arrays** `enemyOrder: number[]` / `towerOrder: string[]` (the worker's live `enemies`/`towers` id sequence). The store builds the reconciled arrays directly from these — see the "Explicit order, not store-side reconstruction" note below. This is cheaper and far less fragile than reconstructing order on the main thread.
- Keep the existing in-memory `SimulationSnapshot` (required `enemies`/`towers`) as the reconciled shape the store holds and consumers read. It is structurally `WireCommon & { enemies; towers }` (full mode minus nothing) — consumer code (`getLatestSnapshot`, render managers, `TowerPanel`) is unchanged.
- `spawnStates` stays full every frame (fixed tiny count); `meta`, `paths`/`pathsVersion`, `particleSpawns`, `lightningEffects`, `stunEffects` unchanged (they live on `WireCommon`).
- Bump `schemaVersion` to **2** (defensive mismatch guard in the store). Both the worker serializer and the main-thread store bump together within a single build, so the guard only ever fires on a genuine version skew (stale worker vs. new main), never in normal play.

## Protocol — `src/sim/WorkerProtocol.ts`
- `{ type: "snapshot"; snapshot: WireSnapshot }` (only the payload type changes; `snapshotAck`/backpressure gate untouched).

## Serializer — `src/sim/SnapshotSerializer.ts`
- Read last-posted state from the engine: `lastPostedEnemies: Map<number, EnemySnapshot>`, `lastPostedTowers: Map<string, TowerSnapshot>`.
- `buildSnapshot` chooses mode:
  - **full** when the maps are empty (baseline / post-`_initMap`) — seeds the main-thread cache.
  - **delta** otherwise.
  - On a **delta**, also emit `enemyOrder` / `towerOrder` as the live `engine.enemyManager.enemies.map(e => e.id)` / `engine.towerManager.towers.map(t => t.id)` sequences. These are the authoritative order the store rebuilds from (see Store reconciliation note). This is cheap: just id arrays, no snapshot objects.
- **Allocation-free comparison** (the key to cutting worker alloc): walk live entities and compare against the stored last-posted snapshot without building a new object. Only build `snapshotEnemy`/`snapshotTower` when a field differs; otherwise skip (no transfer, no alloc).
  - Add `enemySnapshotEqualsLive(e, prev)` / `towerSnapshotEqualsLive(t, prev)` comparing the primitive fields + `statusEffects` length/kind/remaining/magnitude + theme-ref identity.
  - **Exclude** `hitFlash` and `walkingFrameIndex` from comparison (hardcoded to `0` in `snapshotEnemy`).
  - **Correctness constraint (render proxies cache last-written values).** The SVG render proxies only call `setAttribute`/`style` assignment when the new value differs from a per-proxy cached value (`TowerManager.ts:146,191`, `EnemyManager.ts:92,114`, plus the `lastColor`/`lastSpriteId` caches). Therefore a delta that *omits* a tower/enemy whose rendered field changed will leave that entity visibly stale on the main thread, because the proxy sees the unchanged (stale) snapshot value and skips the write. So the equality check MUST compare **every field the renderers read**, or behavior silently diverges from full mode. Enumerate explicitly:
    - **Tower fields to compare:** `level`, `variant`, `angle`, `x`, `y`, `tileX`, `tileY`, `isGhost`, `color`, `animation`, `fireAnimTime`, `health`, `maxHealth`, `base.fixedAim`, `targeting`, `cooldown`, `totalDamageDealt`, `waveDamage`, `placedAt`, `type`. (Rationale: `TowerManager.sync` reads `level/x/y/isGhost/color/fireAnimTime/animation/type/angle`; `UiOverlayManager` tower HP bars read `maxHealth/health/x/y`; `EffectManager` range circle / upgrade button / splash anchor read `level/x/y/tileX/tileY/type`; and `SnapshotStore.capturePreviousWaveDamage` reads `waveDamage` for the per-tower previous-wave `StatsPanel` value, so a lagging `waveDamage` would corrupt that readout even though no renderer draws it directly.)
    - **Enemy fields to compare:** `type`, `x`, `y`, `radius`, `angle`, `hp`, `maxHp`, `shield`, `maxShield`, `slowFactor`, `slowTimer`, `burnTimer`, `onPathBlocked`, `removed`, `gameSeconds`, `hitAnimTime`, `attackAnimTime`, `level`, `statusEffects`, `walking`, `hitReaction`, `attackAnimation`. (EnemyManager reads `walking/radius/angle/x/y/hitReaction/attackAnimation/gameSeconds/hitAnimTime/attackAnimTime/slowFactor/type`; UiOverlayManager reads `x/y/hp/maxHp/shield/maxShield/type`.)
    - Comparing all primitive fields (rather than the smaller "rendered-only" set) is the safe default: firing/idle bookkeeping fields like `cooldown`/`totalDamageDealt` are then also covered, at the cost that a tower ticking `cooldown` down cannot be skipped — which matches today's behavior (every tower is rebuilt every frame today), so no regression. The savings come from *truly static* towers (no fire, no damage, no level/selection change), which keep all these fields constant.
  - Short-circuit `statusEffects` on the frozen `EMPTY_STATUS_EFFECTS` reference identity before deep-comparing entries. Note: the equality check must evaluate "does the live enemy have effects?" **without calling `snapshotEnemy`** — i.e. duplicate the `hasEffects` guard (`e.slowFactor < 1 || e.stunTimer > 0 || maxBurnRemaining > 0 || e.shield > 0 || e.markTargetMult > 0`) against the live `Enemy`. This is logic duplication from `SnapshotSerializer.buildEnemyStatusEffects`; keep it localized and commented so a future status-effect kind change updates both sites.
- **Selected tower is always force-included** in `towersUpdated`, bypassing equality: the tower whose `id === runState.selectedTowerId` **and** the previously-selected id when selection changed this post. Rationale: `snapshotTower` computes derived UI fields (`canUpgrade`, `sellValue`, `stats`, `levelCosts`, `upgradeCostAt5`, `milestoneBonus`) only for the selected tower, and several depend on `gold`, which changes without any change to the tower's own primitives — equality alone would leave `TowerPanel` stale. Track `lastPostedSelectedTowerId` on the engine to detect the change. **Edge case:** if the previously-selected tower was *sold* in the same post, it is already in `towersRemoved` and gone from `lastPostedTowers`; skip force-including it (selection is cleared on sell, so this is the normal case).
- `removed` = ids in `lastPosted*` not present in the current live set.
- **Update `lastPosted*` + `lastPostedSelectedTowerId` inside `buildSnapshot` itself** (set updated entries, delete removed, record the posted `selectedTowerId`). This is safe because `buildSnapshot` is only ever invoked on the *posted* path in `WorkerEntry.ts` (the two call sites at `:135` and `:164` are both inside the non-dropped branch), so dropped backpressure ticks never reach it and never mutate the cache. This also lets the unit test drive `full → delta` by calling `buildSnapshot` twice on one engine without the worker loop.

## Engine — `src/sim/GameEngine.ts`
- Add `lastPostedEnemies: Map<number, EnemySnapshot>`, `lastPostedTowers: Map<string, TowerSnapshot>`, `lastPostedSelectedTowerId: string | null`.
- Clear all three in `_initMap` (the single funnel for both `loadMap` and `loadRandomMap`), alongside the existing `lastPostedPathVersion = 0` / `lastPostedWaveGraphGeneration = 0` / `gridLayoutEnabled = true` reset at `GameEngine.ts:181-183` — so the next post after a (re)init is a forced **full**.

## Store reconciliation — `src/sim/SnapshotStore.ts`
- `apply(wire: WireSnapshot)`:
  - **schemaVersion guard**: if `wire.schemaVersion !== EXPECTED (2)`, `console.warn` and drop the snapshot (`current` unchanged; do not throw — must not kill the render loop). NOTE: this means every hand-built test snapshot must set `schemaVersion: 2` (see Tests).
  - **Reconcile to a `SimulationSnapshot` first**, then run the **existing pipeline unchanged**: `capturePreviousWaveDamage`, the `waveGraphDots` merge, `resolveSelectedTower`, and `mirrorToGameStore` (SnapshotStore.ts:100-229). The reconciled object is assigned to `this.current` and `latestSnapshot` exactly as today — this is what keeps `getLatestSnapshot`, the render managers, and `TowerPanel` byte-for-byte compatible with current behavior. Only the *construction* of `current` differs (full vs. merge); everything downstream is untouched.
  - **full** → build `current` directly from `enemies`/`towers` (copy the arrays/refs as today); seed `enemyOrder`/`towerOrder` from `enemies.map(e => e.id)` / `towers.map(t => t.id)`.
  - **delta** → merge into `current` from the **explicit `enemyOrder`/`towerOrder` id arrays** the worker shipped (do NOT reconstruct order on the main thread — see note below):
    - Build a lookup `Map<id, updatedSnapshot>` from `*Updated`.
    - Walk `enemyOrder`/`towerOrder`; for each id, take the updated snapshot if present, else keep the prior `current` entry; drop ids present in `*Removed`. This yields the reconciled `enemies`/`towers` arrays in the worker's authoritative order (reuse the array object where possible to avoid per-frame realloc).
  - **Explicit order, not store-side reconstruction.** The earlier design maintained `enemyOrder`/`towerOrder` on the main thread via "upsert by id + append new + splice removed," relying on an invariant that the sim "only appends / removes-in-place, never reorders." That invariant is **false**: `TowerManager.towers` is reordered by `.filter` on sell (`TowerManager.ts:176,185`) and `EnemyManager.enemies` by `.splice` on death (`EnemyManager.ts:123`). The append+remove reconstruction *happens* to reproduce the same id-order as that recompaction only because both are relative-order-preserving on survivors — correct today, but silently fragile to any future reorder (e.g. re-sorting towers on upgrade). Shipping the worker's authoritative id-order in every delta removes the dependency entirely and makes the integration test's "identical to a full rebuild" claim trivially true. Cost is a tiny id array per delta.
  - **Order is not a rendering-correctness constraint today, but must still be preserved.** The HP-bar / tower-bar pools are assigned *positionally* and recompute each bar's `transform` from the entity's own `x/y` every frame (`UiOverlayManager.ts:213-284` enemies, `:305-324` towers) — the `hpLastTransform` cache only avoids redundant `setAttribute` calls, it does not pin a bar to a specific tower. So a divergent reconciled order would NOT draw a bar over the wrong tower. **However**, shipping authoritative `enemyOrder`/`towerOrder` is still required for two reasons: (1) it keeps the reconciled array *identical* to full-mode order, which is exactly what the integration test asserts, and (2) it stays robust to any future change that *does* make order semantically meaningful (e.g. a pool that keys by stable slot). Treat `towerOrder` and `enemyOrder` with equal rigor.
  - **Defensive**: if a delta arrives with no `current` (shouldn't happen due to baseline), treat `*Updated` as the initial set and build order from the shipped `enemyOrder`/`towerOrder` (fall back to `*Updated` id order if those are absent).
- `mirrorToGameStore`, `resolveSelectedTower`, `getLatestSnapshot` unchanged (read the reconciled full arrays).
- **Add a comment** at `apply` (and/or on the `WireSnapshot` type) documenting that this is the single reconciliation point: **future stream consumers** (replay recorder, network relay, analytics) should subscribe here — either to the raw `WireSnapshot` delta stream (bandwidth-sensitive consumers) or to the reconciled full `SimulationSnapshot` (`getLatestSnapshot`) for full-state consumers. This multi-consumer readiness is part of the motivation.

## Worker — `src/sim/WorkerEntry.ts`
- No structural change; the empty-`lastPosted*` maps force the baseline **full** through the serializer on the first post after `_initMap`. Backpressure/`snapshotAck` logic unchanged. Note the existing invariant that `buildSnapshot` runs only immediately before `postMessage`, so deltas are always computed against the last *posted* state (dropped ticks never mutate `lastPosted*`).

## Scope notes / non-goals
- Enemy position still ships every post (they move); real savings = static towers dropped + dead-enemy removals as bare ids. **Measure the clone-size reduction post-implementation** to confirm it justifies the reconciliation complexity before ruling the deferred lever back in.
- Typed-array packing / zero-copy transfer for enemies is explicitly **out of scope** here; revisit if transfer cost remains hot.
- No change to commands, `HostBindings`, or the main-thread render loop in `SvgGameRoot.vue` beyond `SnapshotStore.apply` (already called).
