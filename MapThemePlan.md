# Map Theme System Plan

## Goal & Scope

Introduce a **Map Theme** system that swaps the *visual identity* of towers, enemies, and map tiles on the `/game` screen, while leaving every gameplay stat, effect, overlay, and every non-`/game`/`/map-select` screen untouched. The current polygon-based art becomes the first/default theme (id `default`).

A theme is selected on the `/map-select` screen via a drop-down. The selected theme is locked in for that run — no in-game theme switching.

### In scope (themeable, `/game` only)
- **Towers**: SVG sprite frames (`<symbol>` inner content), primary color, icon character, display name.
- **Enemies**: SVG sprite frames for walking + hit-reaction (`<symbol>` inner content), primary color, shape label, display name.
- **Maps**: per-region display name + 5 tile images per region (1 path tile + 4 terrain-height tiles) + 1 base 3×3 SVG image per region.

### Out of scope (explicitly NOT themeable)
- All gameplay/functional data: tower cost/range/damage/fire rate/variants/milestones; enemy HP/speed/bounty/shield/heal/resist; map layout, region difficulty, gem multipliers, wave composition.
- All effects & overlays: particles, lightning, stun auras, build-preview rect, range circles, upgrade button, HP/shield bars, boss HP text, hit-flash logic, slow filters, spawn markers, path polyline highlights, blocked-tile cross-hatch, background fill, terrain height-number text, tile border stroke.
- All screens except `/game` and `/map-select` (menu, skill-tree, end screens, history). The Skill Tree always uses the `default` theme's tower data.
- In-game theme switching, audio (`SoundManager`), new themes beyond `default`.

## Key Constraints & Invariants

1. **Stable symbol ID contract.** Symbol IDs stay exactly `enemy-${type}-f${frameIndex}`, `enemy-${type}-hit-f${frameIndex}`, `tower-${type}-f${frameIndex}`. A theme swaps only the *content* inside those `<symbol>`s and the color/name/icon metadata. The render managers (`render/svg/EnemyManager.ts`, `render/svg/TowerManager.ts`) need **zero changes**.
2. **Variable frame counts.** The number of walking and hit-reaction frames may differ per enemy and per theme. The render managers already read `referenceImages.length` dynamically (`computeEnemyFrame`, `computeTowerFrame`, hit-reaction indexing), so this works as-is. Symbol generation must iterate the theme's frame arrays, not hard-coded counts.
3. **Stats never come from the theme.** `TOWER_BASE`, `TOWER_VARIANTS`, `ENEMY_TYPES` numeric fields (`baseHp`, `speed`, `bounty`, `radius`, `shield`, `heal`, `healRange`, `resist`, `slowResist`), `MAP_LEVELS`, gem multipliers, boss cadence — all stay in the constants files, unchanged. Only *visual/displayed* fields move to the theme JSON.
4. **`regionId` still drives map generation.** `Map.ts`/`Grid.ts`/`WaveManager.ts` continue to receive `regionId` and behave identically. Only the *tile images* and *displayed region name* differ per theme.
5. **Default theme is the polygon art.** Its JSON is generated by extracting the current visual fields from `ConstantsTower.ts`, `ConstantsEnemy.ts`, and `Constants.ts` into `default-map-theme.json`.

### Image field format
Every `image` value is **either**:
- **Inline SVG text** — a complete `<svg>...</svg>` document (wrapper is stripped by the existing `stripSvgWrapper` helper before being placed inside `<symbol>` or tile `<g>`). Used as-is.
- **A path or URL** — e.g. `"./themes/data/default/tiles/region0/path.svg"` or `"https://.../x.svg"`. The theme loader (see below) resolves these at load time by `fetch()`-ing the resource, reading the SVG text, and replacing the field with the inlined string. This normalization runs once per theme load; downstream code (`useSvgStaticContent`, `Tower`/`Enemy` constructors) always sees inlined SVG strings.

For the `default` theme all images are inline SVG (the current polygon art), so no fetching is needed.

### Tile image contract
- Tile images render into a 36×36 (TILE_SIZE) box. Their `<svg viewBox>` should be `0 0 36 36` (or any 1:1 ratio; the loader scales via `<use>` width/height).
- A tile image replaces the **fill rect** and the **terrain diagonal cross-hatch decoration** only. The following overlays are drawn on top by `useSvgStaticContent` and are **not** part of the tile image: tile border stroke, terrain height-number text, blocked-tile cross-hatch, spawn marker, path polyline highlight, background fill. (These are listed as out-of-scope overlays.)
- The 5 tile images per region: `path` (used for path + spawn tiles' base fill), `terrain1`–`terrain4` (one per height level 1–4).

### Base 3×3 image contract
- One SVG per region, rendered at the 3×3 base footprint (`size*2.7` × `size*2.7` currently, centered on the base tile). Replaces the procedural `renderBaseSvg` polygon output entirely. The `base-gradient` `<linearGradient>` in `mapDefsContent` stays (themes may reference `url(#base-gradient)` inside their base SVG, or not).

### Tower/enemy frame contract
- Tower frames: `<svg viewBox="-16 -16 32 32">…</svg>` (matches current). Placed in `<symbol id="tower-${type}-f${i}" viewBox="-16 -16 32 32">`.
- Enemy frames: `<svg viewBox="-1 -1 2 2">…</svg>` (matches current). Placed in `<symbol id="enemy-${type}-f${i}" viewBox="-1 -1 2 2">` and `enemy-${type}-hit-f${i}`.
- Frame arrays may be any length ≥ 1. Per-frame `duration` is optional (current render code uses only the aggregate `duration`); keep the field for forward-compat.

## Theme Registry & Loader

New module: `src/render/themes/index.ts`

```ts
export interface MapThemeManifestEntry { id: string; label: string; }
export const MAP_THEME_MANIFEST: MapThemeManifestEntry[] = [
  { id: "default", label: "Polygon (Default)" },
  // future themes appended here only
];
export const DEFAULT_THEME_ID = "default";

// Lazy loaders. Default uses a static import (see "Default handling" below)
// so it is available synchronously for non-game screens (SkillTree).
const loaders: Record<string, () => Promise<MapThemeData>> = {
  default: () => import("./data/default-map-theme.json").then((m) => m.default as MapThemeData),
  // future:  polygon: () => import("./data/polygon-map-theme.json")...
};
export function getThemeLoader(id: string): (() => Promise<MapThemeData>) | undefined {
  return loaders[id];
}
```

A second theme is added later by: dropping a `<id>-map-theme.json`, adding one `{id,label}` to `MAP_THEME_MANIFEST`, and one entry to `loaders`. Nothing else changes.

### Default handling
The `default` theme JSON is `import`-able (Vite supports JSON imports). To keep non-game screens (SkillTree, etc.) synchronous and simple, the theme store **preloads** the default theme at app init (see below) and caches it. The active theme for `/game` is loaded on demand. If the user selects `default` in MapSelect, the loader returns the already-cached default.

The spec says "lazy-loaded JSON file"; the `default` file is still a JSON file fetched via dynamic `import()` (code-split), it is just *also* preloaded eagerly so the SkillTree never has to await. Non-default themes are truly lazy (only loaded when selected).

## Runtime Access: `useMapThemeStore` (Pinia)

New store: `src/stores/mapTheme.ts`

State:
- `defaultTheme: MapThemeData | null` — resolved at app init.
- `activeThemeId: string` — selected theme id for the next/current run. Defaults to `"default"`.
- `activeTheme: MapThemeData | null` — resolved theme data for `activeThemeId`. Mirrors `defaultTheme` when id is `default`.
- `loading: boolean`, `error: string | null`.

Actions:
- `async preloadDefault()` — called once from `main.ts` before `app.mount`. Uses a synchronous JSON import (`import("./data/default-map-theme.json")`) so `defaultTheme` is available immediately with no race condition — no loading gate needed. Sets `defaultTheme` and, if `activeThemeId === "default"`, also `activeTheme`.
- `async loadActive(id)` — sets `activeThemeId`, resolves the theme via `getThemeLoader(id)`, normalizes external image refs to inline SVG, sets `activeTheme`. Returns the resolved data.
- `getTowerVisual(typeId): TowerVisualMeta | null` — reads `activeTheme ?? defaultTheme`. Used by `Tower` constructor and `/game` UI components.
- `getEnemyVisual(typeId): EnemyVisualMeta | null` — same.
- `getRegionVisual(regionId): RegionVisualMeta | null` — same.
- `getDefaultTowerVisual(typeId)` / `getDefaultEnemyVisual(typeId)` — always read `defaultTheme`, for non-game screens.

Getters:
- `availableThemes: MapThemeManifestEntry[]` — re-export of `MAP_THEME_MANIFEST` for the MapSelect drop-down.
- `activeThemeLabel: string` — for display.

- **Enemies**: for each of `minion|runner|tank|shielded|healer|boss`, copy `name`, `color`, `shape`, and `walking.referenceImages[].svg` → `walking.frames[].image`, `hitReaction.referenceImages[].svg` → `hitReaction.frames[].image`. Copy `duration`.
- **Regions** (3): copy `name` from `Regions[i]`. For tile images, generate inline SVGs that reproduce the current `getTileSvg` fill + terrain cross-hatch:
  - `path` tile: `<svg viewBox="0 0 36 36"><rect width="36" height="36" fill="${pathColor}"/></svg>` (terrain cross-hatch not applicable to path).
  - `terrain1..4`: `<svg viewBox="0 0 36 36"><rect width="36" height="36" fill="${heightColors[h-1]}"/><path d="M7.2,7.2 L28.8,28.8 M28.8,7.2 L7.2,28.8" stroke="rgba(0,0,0,0.12)" stroke-width="0.5"/></svg>` (reproduces current fill + diagonal cross-hatch; the height-number text and border are drawn as overlays by `useSvgStaticContent`, so not in the image).
  - `base`: inline SVG reproducing `renderBaseSvg` output (the rounded-corner path + gem circles + hex emblem). This is the polygon base art.

A one-off generator script (`scripts/generate-default-theme.ts`, run via `npx tsx`) can produce this JSON from the current constants to avoid hand-copying 100+ frames. The script is throwaway but committed for reproducibility. After generation, the constants files are slimmed (visual fields removed).

## Non-game Screen Data Flow

- **SkillTree.vue** reads `mapThemeStore.defaultTheme.towers[id]` for name/color/icon. `defaultTheme` is preloaded at app init, so this is synchronous and stable.
- **HistoryScreen.vue** / **EndScreen.vue** read `mapThemeStore.defaultTheme.regions[i].name` for region display.
- **MainMenu.vue** has no theme dependency.
- No non-game screen ever reads `activeTheme`. This guarantees the Skill Tree tower data is invariant regardless of the active run's theme (per spec).

## MapSelect Drop-down & Responsive Switching

- The drop-down lists `mapThemeStore.availableThemes` (from `MAP_THEME_MANIFEST`). Only `default` exists for now; the drop-down is still added so the system is wired for future themes.
- On `change`, `mapThemeStore.loadActive(id)` runs. Because `default` is preloaded, switching to/from default is instant. Future themes fetch + normalize on first select (a brief loading indicator on the grid is acceptable).
- `mapEntries` and `mapsByRegion` are `computed` over `mapThemeStore.activeTheme`, so region names and map display names update reactively when the theme resolves.
- The grid uses `grid-template-columns: repeat(auto-fill, minmax(140px, 1fr))` (changed from `repeat(6, 1fr)`) so it is fully responsive — 1 column on narrow screens, up to 6+ on wide. The drop-down sits in the header next to "Select Map" / "Back".
- `startMap` awaits theme resolution before navigating to `/game`.

## In-Game Theme Wiring (per run)

1. User picks theme in MapSelect → `mapThemeStore.activeTheme` resolved.
2. `startMap` sets `gameStore.map`/`mapIndex`, navigates to `/game`.
3. `SvgGameRoot.onMounted` constructs `new GameEngine(gameStore, persistStore, mapThemeStore.activeTheme)`.
4. `GameEngine._initMap` constructs `TowerManager`/`EnemyManager` with the theme; they pass `theme.getTowerVisual(type)`/`theme.getEnemyVisual(type)` to `Tower`/`Enemy` constructors.
5. `useSvgStaticContent(map, grid, activeTheme)` builds `<symbol>`s and tile strings from the theme.
6. Render managers read stable IDs + per-entity visual fields — unchanged.

Theme is immutable for the run; `activeTheme` does not change while on `/game`.

## Symbol ID Stability & Variable Frame Counts

- IDs: `tower-${type}-f${i}`, `enemy-${type}-f${i}`, `enemy-${type}-hit-f${i}` — unchanged contract.
- `buildSymbolsFromTheme` emits exactly as many symbols as the theme's `frames.length` for each animation. The render managers' `computeEnemyFrame`/`computeTowerFrame` use `refImages?.length || 1` and modulo, so a 4-frame theme and an 8-frame theme both animate correctly without code changes.
- Hit-reaction indexing in `EnemyManager` uses `hitReaction.referenceImages.length` — variable, works.
- `EnemyRenderProxy`/`TowerRenderProxy` read `enemy.walking`/`tower.meta.animation` (now populated from theme) — field shape unchanged (`{ duration, referenceImages: { image }[] }` or we keep the name `referenceImages` for zero-touch). **Recommendation: keep the field name `referenceImages` and frame field `{ svg }` in the runtime `MapThemeData` (post-normalization) to avoid touching render managers.** The `normalizeThemeImages` function maps each JSON `frames[].image` → runtime `referenceImages[].svg` (one entry per frame, preserving the original `duration` on the parent animation config). This makes the render managers truly zero-change.
